# Phase 5 ‚Äî UI Design Contract

> Visual and interaction contract for Phase 5: Email Capture and Export.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> Phase 5 adds three UI surfaces to the existing app: (1) a persistent
> chrome email button visible on every screen, (2) the email sign-up
> screen with GDPR consent form and post-submit confirmation view, and
> (3) a CSV export section inside the existing admin panel. No new
> screens are introduced ‚Äî `#/email` is the only new route. All design
> tokens are inherited from Phase 1 and remain unchanged.

| sm | 8px | Input internal padding (vertical), label-to-input gap |

- Email input: 48px minimum height (`min-height: var(--touch-target-min)`), same as all other inputs in the project.
- Submit button: 56px height when using `.btn-large` modifier ‚Äî matches existing `btn-large` definition.

## Screen Specifications

### New Surface 1: Chrome Email Button

**Location:** `index.html` global chrome layer, fixed position.

**Position:** Bottom-left. `position: fixed; bottom: var(--space-md); left: var(--space-md)` (16px from both edges). This is the only available corner ‚Äî top-left is occupied by `#chrome-home`, bottom-right and right side are occupied by `#chrome-qr`.

**Dimensions:** 48 √ó 48px, matching `#chrome-home` exactly.

**Visual:**
- Background: `rgba(14, 27, 77, 0.85)` ‚Äî same translucent navy as `.chrome-home`
- Border: `1px solid var(--color-accent)` ‚Äî same as `.chrome-home`
- Border radius: 8px ‚Äî same as `.chrome-home`
- Icon: Unicode ‚úâ (`&#9993;`) at 22px ‚Äî slightly smaller than the 24px home glyph to account for the envelope character's wider visual footprint
- Icon colour: `var(--color-accent)`
- z-index: 100 ‚Äî same layer as `#chrome-home` and `#chrome-qr`

**Pressed state:** `opacity: 0.8` on `:active` ‚Äî same pattern as `.chrome-home:active`

**Aria label:** `"Sign up for email updates"` ‚Äî describes the destination, not the icon

**CSS class:** `.chrome-email`

Source: CONTEXT.md D-01, D-02, D-03; RESEARCH.md ¬ßPattern 1; main.css `.chrome-home` (lines 284‚Äì305)

### New Surface 2: Email Sign-Up Screen (`#/email`)

**Triggered when:** User taps the chrome email button from any screen. Route: `#/email`. Renders via `renderEmail()` in `src/email.js`.

**Layout (landscape viewport 1366 √ó 1024 CSS px):**

The screen centres a single form card vertically and horizontally. The form card must not be full-viewport-width ‚Äî it should feel like a focused panel, not a sprawling form.

- Screen: `display: flex; flex-direction: column; align-items: center; justify-content: center;` ‚Äî inherits `.screen` base class
- Screen background: `--color-bg` (`#0d0f1a`)
- Form card: `background: var(--color-surface)`, `border-radius: 12px`, `padding: var(--space-xl)` (32px), `width: 100%; max-width: 560px`
- Screen must have `overflow-y: auto` to allow scroll if the keyboard pushes content ‚Äî mitigates iOS keyboard overlay pitfall

**Form card contents, top to bottom:**

1. **Screen heading:** "Sign Up for Updates"
   - Inter 28px weight 700, `--color-text-primary`, `line-height: 1.2`
   - `margin-bottom: var(--space-md)` (16px)

2. **Sub-heading / instruction:** "Get early access to new designs and exclusive event discounts."
   - Inter 16px weight 400, `--color-text-secondary`, `line-height: 1.5`
   - `margin-bottom: var(--space-lg)` (24px)

3. **Email input:**
   - `<input type="email" inputmode="email" autocorrect="off" autocapitalize="off" spellcheck="false">`
   - Width: 100% of form card
   - Height: 48px minimum (`min-height: var(--touch-target-min)`)
   - Padding: `var(--space-sm) var(--space-md)` (8px vertical, 16px horizontal)
   - Background: `var(--color-bg)`, border: `1px solid var(--color-text-secondary)`, border-radius: 4px
   - Text: Inter 16px weight 400, `--color-text-primary`
   - Focus state: `border-color: var(--color-accent)`, no outline (same as `.admin-input:focus`)
   - Placeholder text: "your@email.com" ‚Äî `--color-text-secondary`
   - `margin-bottom: var(--space-sm)` (8px)

4. **Inline validation error (conditional):**
   - Shown below the input when submit is tapped with an invalid email
   - Text: "Please enter a valid email address." ‚Äî Inter 14px weight 700, `--color-destructive`
   - `margin-bottom: var(--space-sm)` (8px)
   - Hidden (`display: none`) until validation fails; shown inline, does not shift layout unexpectedly ‚Äî reserve space with `min-height: 20px`

5. **GDPR consent group:**
   - Wrapping `<label>` element ‚Äî the entire row (checkbox + text) is the tap target
   - Min-height: 48px, `display: flex; align-items: center; gap: var(--space-sm)` (8px)
   - Checkbox: `<input type="checkbox">` ‚Äî browser default rendering, unchecked by default; do NOT set `checked` attribute
   - Checkbox visual size: 20 √ó 20px minimum; use `width: 20px; height: 20px; flex-shrink: 0` to prevent squishing
   - Label text: "I agree to receive email updates from The ID Card Factory." ‚Äî Inter 16px weight 400, `--color-text-primary`, `line-height: 1.5`
   - The exact wording above is mandatory ‚Äî GDPR compliance (CONTEXT.md D-05)
   - `margin-bottom: var(--space-lg)` (24px)

6. **Submit button:**
   - Text: "Sign me up"
   - Reuses `.btn-primary.btn-large` ‚Äî full-width (100%), 56px height, `--color-accent` background, `--color-bg` text, Inter 16px weight 700
   - `disabled` attribute present on initial render; removed only when checkbox `change` event fires with `checked === true`
   - Disabled state: `opacity: 0.5; cursor: not-allowed` ‚Äî existing `.btn-primary:disabled` rule applies
   - Pressed state (when enabled): `opacity: 0.8` on `:active`

**Idle timer:** The 3-minute grace period for `#/email` is already wired by `initEmailGracePeriod()` in `idle.js` ‚Äî no additional idle handling required in `email.js`.

**Global chrome:** `#chrome-home`, `#chrome-qr`, and `#chrome-email` remain visible on this screen (they are in the global chrome layer above `#app`). The chrome email button should visually indicate the current screen ‚Äî apply `opacity: 0.5` to `#chrome-email` when the current hash is `#/email` to signal it is already active. Wire this state in `app.js` on hashchange.

Source: CONTEXT.md D-04, D-05, D-06, D-07; RESEARCH.md ¬ßPattern 2, ¬ßPitfall 2, ¬ßPitfall 3; REQUIREMENTS.md EMAIL-01, EMAIL-02

### New Surface 3: Email Confirmation View

**Triggered when:** `dbAdd('emails', record)` resolves successfully. Replaces the form content inside `#screen-email` in-place ‚Äî no route change, no new DOM container (CONTEXT.md D-11).

**Layout:**

The confirmation view fills the same form card area as the sign-up form. The card content is replaced entirely.

- Form card background and dimensions unchanged (`--color-surface`, `max-width: 560px`)
- Contents, top to bottom:

1. **Success icon:**
   - Unicode checkmark: ‚úì (`&#10003;`) ‚Äî rendered at 48px, `--color-accent`, `text-align: center`
   - Do NOT use an SVG or external asset ‚Äî Unicode glyph is sufficient and has zero cache overhead
   - `margin-bottom: var(--space-md)` (16px)

2. **Confirmation heading:** "You're in!"
   - Bebas Neue 48px, `--color-text-primary`, `line-height: 1.1`, `text-align: center`, `text-transform: uppercase`
   - `margin-bottom: var(--space-sm)` (8px)

3. **Confirmation body:** "Thanks for signing up. We'll let you know about new designs and event exclusives."
   - Inter 16px weight 400, `--color-text-secondary`, `line-height: 1.5`, `text-align: center`
   - `margin-bottom: var(--space-sm)` (8px)

4. **Signup count (optional, async):** "You're #N to sign up at this event!"
   - Inter 14px weight 400, `--color-text-secondary`, `line-height: 1.5`, `text-align: center`
   - Loaded asynchronously via `dbGetAll('emails')` filtered to current event; rendered once the promise resolves
   - Initially hidden; inserted into the DOM and made visible when the count resolves
   - If count fetch fails, this line is silently omitted ‚Äî it is a nice-to-have, not critical
   - `margin-bottom: var(--space-lg)` (24px)

5. **Countdown text:** "Returning to catalogue in 5‚Ä¶"
   - Inter 16px weight 400, `--color-text-secondary`, `line-height: 1.5`, `text-align: center`
   - Updated in-place as the counter ticks: "‚Ä¶in 4‚Ä¶", "‚Ä¶in 3‚Ä¶", "‚Ä¶in 2‚Ä¶", "‚Ä¶in 1‚Ä¶"
   - Update via `textContent` on the countdown element ‚Äî do NOT re-render the whole view
   - `margin-bottom: var(--space-md)` (16px)

6. **Early-dismiss button:** "Back to browsing"
   - Reuses `.btn-secondary.btn-large` ‚Äî full-width, 56px height, transparent background, `--color-text-secondary` text and border
   - Tapping this clears the countdown interval and immediately navigates to `#/`

**Entry transition:**
- The form card content replacement is instantaneous ‚Äî no fade or slide. The `.fade-in` class from `animations.css` is NOT applied here. Immediate render is correct: the user just submitted and should see the result without delay.

Source: CONTEXT.md D-08, D-09, D-10, D-11; RESEARCH.md ¬ßPattern 2 ¬ßOpen Question 2

### Modified Surface 4: Admin Panel ‚Äî Email Export Section

**Location:** Inside `renderAdminPanel()` in `admin.js`. New `admin-section` div inserted before the Exit button, after the existing Sync Status section.

**Visual:** Matches all existing `admin-section` cards exactly ‚Äî same `--color-surface` background, `border-radius: 8px`, `padding: var(--space-lg)`.

**Section heading:** "Email Export"
- `.admin-section-heading` ‚Äî Inter 18px weight 700, `--color-text-primary` (existing class, unchanged)

**Export button:**
- Text: "Export / Share Emails (CSV)"
- The label explicitly says "Share" because iPadOS Safari presents a share sheet rather than a silent download. This sets correct admin expectations (RESEARCH.md ¬ßPitfall 4).
- Reuses `.btn-primary.btn-large` ‚Äî full-width, 56px height
- `margin-top: var(--space-md)` (16px below heading)

**Status message (conditional):**
- Zero emails state: "No emails captured for this event yet." ‚Äî `.sync-result-detail` class ‚Äî Inter 14px weight 400, `--color-text-secondary`. Shown inline below button after tap; button remains tappable.
- Export success state: "Exported N email(s)." ‚Äî `.sync-result-detail` class ‚Äî same style. Shown inline below button after successful download.
- Both states use the existing `.sync-result-detail` CSS class ‚Äî no new CSS needed.
- Status message element is rendered at mount (`display: none`), shown on tap result ‚Äî avoids layout shift.

Source: CONTEXT.md D-14, D-15, D-16, D-17, D-18, D-19; RESEARCH.md ¬ßPattern 3; main.css `.admin-section`, `.admin-section-heading`, `.btn-primary`, `.btn-large`, `.sync-result-detail`

### Error State: dbAdd Failure

**Triggered when:** `dbAdd('emails', record)` rejects. This is rare (device storage full, IDB corruption) but must not silently discard the user's submission.

**Location:** Shown inline within the email form ‚Äî below the submit button, replacing the disabled state of the button temporarily.

**Visual:**
- Error message: "Something went wrong saving your details. Please try again." ‚Äî Inter 14px weight 700, `--color-destructive`, `text-align: center`
- Submit button: re-enabled immediately after the error, allowing the user to retry
- Form fields retain their values (email address and checkbox state are not cleared)
- The `record` object (email, eventName, eventDate, consentAt) must be logged to `console.error` for admin diagnostics ‚Äî data must not be silently dropped

Source: CONTEXT.md (Claude's Discretion ‚Äî error state); RESEARCH.md ¬ßPattern 2 `showEmailError()`; REQUIREMENTS.md EMAIL-03
